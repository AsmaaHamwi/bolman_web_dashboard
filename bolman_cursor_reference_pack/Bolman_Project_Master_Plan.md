# بولمان — المرجع الشامل للمشروع  
## دليل الفهم والتطوير لـ Cursor

هذا الملف مرجع تنفيذي لمشروع **بولمان للنقل بين المحافظات**، ويجمع الرؤية، البنية، الجداول، الصلاحيات، المنطق التشغيلي، واجهات الويب والموبايل، والسيناريوهات الحرجة.

# 2) القرارات الثابتة

## 2.1 الهوية والمصادقة
- `users.id` في `public` مرتبط بـ `auth.users.id`.
- الاعتماد الرسمي على Supabase Auth.
- الإنشاءات الإدارية الحساسة تتم عبر Edge Functions أو RPC آمنة.
- لا يوجد Register في الداشبورد.
- Register موجود فقط في تطبيق الراكب.

## 2.2 الصلاحيات
- لا يوجد جدول صلاحيات موحد.
- يجب بقاء:
  - `system_staff_permissions`
  - `company_staff_permissions`

## 2.3 التقييم
- التقييم داخل `bookings` فقط.
- تقييم واحد لكل حجز.
- نجوم فقط.
- لا يوجد comment.

## 2.4 الدفع
- `payments` ليست One-to-One مع `bookings`.
- العلاقة الصحيحة: `bookings 1:N payments`.

## 2.5 المقاعد
- عرض المقاعد يعتمد على:
  - `bus_seats`
  - `booking_seats`
  - `seat_locks`
  - `trip_stops`
- استخدم Service مستقلة:
  - `SeatAvailabilityService.getSeatsStatus(trip_id, from_trip_stop_id, to_trip_stop_id)`

## 2.6 تأكيد الحجز
- يجب أن يتم داخل Transaction واحدة.
- أي فشل يعني rollback.

# 17) أهم المشاكل التي يجب أن يحلها Cursor

## 17.1 صفحة الحجوزات فارغة
افحص:
- companyId
- الاستعلام
- RLS
- وجود الحجوزات فعلاً

## 17.2 الحجز اليدوي
لا تجعل عدد الركاب منفصلًا عن المقاعد المختارة.

### الصحيح
```ts
const passengerCount = selectedSeatIds.length;
```

## 17.3 إنشاء الرحلة
- أول stop ثابت
- آخر stop ثابت
- المحطات الوسطية اختيارية
- لا تعدّل أول/آخر stop يدويًا

## 17.4 الباصات
فلترة الباص حسب:
- الشركة
- الحالة
- المدينة الحالية

## 17.5 الترجمة
لا تترك labels إنكليزية في الواجهة العربية.

# 18) ما الذي يجب أن ينتبه له Cursor؟

- لا يضع `service_role` داخل React أو Flutter.
- لا يدمج جداول الصلاحيات.
- لا يطلب `owner_user_id` يدويًا.
- لا ينشئ السائق مرتين.
- لا يجعل `company_id` في bookings لأنه غير موجود.
- لا يجعل عدد الركاب مستقلًا عن المقاعد.
- لا يترك إنشاء الرحلة قابلًا للكسر.
- لا يغير الرحلة بعد وجود حجوزات إلا في offer fields.
